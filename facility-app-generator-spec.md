# 施設検索Webアプリ・ジェネレーター QGISプラグイン 設計書 兼 指示書

## 0. この文書の位置づけ

この文書は2つの役割を兼ねる。

1. **設計書**：人間（りんとゆう）が全体像を確認・見直すための資料
2. **指示書**：QGIS Python Console／庁内AI／Claude Code等に実装を依頼するときに、そのまま渡せる仕様書

そのため、各セクションは「何を作るか（目的）」と「どう作るか（仕様）」をセットで書いている。実装を依頼する際は、該当セクションをそのままコピーして渡せばよい。

## 1. 目的・背景

### 1.1 課題

- `qgis2web` はレイヤーの表示設定をそのままJS/Leafletに変換するだけで、検索・条件抽出・複合的な分析機能を持たせられない（オンオフ程度の操作しかできない）。
- 一方で `osaka-facilities` プロジェクトでは、検索・フィルタ・圏域カバー分析・ヒートマップ等、実用的な機能をLeaflet.js上にすでに実装済み。
- この資産を **他の部署・他のデータセットでも再利用できる形（プラグイン）** にしたい。

### 1.2 コンセプト

> 「JSON書き出しツール」ではなく「施設検索Webアプリ・ジェネレーター」として設計する。

QGIS上で開く**独立したウィンドウ（ダイアログ）**で、

- どのレイヤーを使うか
- どの属性を検索・表示対象にするか
- どの機能（圏内施設抽出（半径検索）／圏域分析／ヒートマップ等）を有効にするか
- 圏域の初期値（施設種別ごとの距離）

を **GUIで設定** し、ボタン一つで書き出す。出力形式は**デフォルトで「HTML＋外部ファイル（GeoJSON/config.json）に分ける」分割ファイル方式**とし、必要に応じて「単一HTMLファイルに埋め込む」方式もオプションとして**選択できる**ようにする（詳細は3.1・5.3参照）。いずれの形式でもサーバー不要、ブラウザで開くだけ、あるいはGitHub Pages等に置くだけで動作する。

## 2. 全体アーキテクチャ

役割を明確に分離する。プラグインが複雑なJS描画ロジックを抱え込むと保守できなくなるため、以下の分担を厳守する。

| 部分 | 役割 | 実装言語 |
|---|---|---|
| **QGIS Plugin（設定・書き出し担当）** | レイヤー選択、属性マッピング、機能ON/OFF設定、圏域パラメータ設定、**QGISレイヤーの表示設定（シンボル・ラベル）の抽出**、GeoJSON生成、設定JSON・スタイルJSON生成、HTMLテンプレートとの合体・出力 | Python (PyQGIS)　※UI部分は「Qt Designerで.ui作成」または「PyQtコードのみで組む」のどちらでも可（庁内環境でQt Designerが使えない場合はコードのみ方式を選択） |
| **HTMLテンプレート（表示・機能担当）** | 地図描画（**抽出したスタイルJSONに基づくマーカー・ラベル描画**）、検索、フィルタ、圏内施設抽出（半径検索）、圏域分析、ヒートマップ等 | HTML/CSS/JavaScript (Leaflet.js) ※osaka-facilities資産を流用・モジュール化 |
| **設定JSON（橋渡し）** | プラグインが生成した設定を、テンプレート側のJSが読み込んで機能のON/OFFや表示項目を制御する | JSON |
| **スタイルJSON（橋渡し・新設）** | QGISレイヤーの表示設定（マーカーの色・形・サイズ、ラベルのフォント・サイズ・色・バッファ、線の太さ・色等）をJSに渡すための中間形式 | JSON |
| **データGeoJSON** | 施設データ本体（属性・位置情報） | GeoJSON |

### 2.1 処理の流れ（ビルドフロー）

```
[QGIS上でレイヤー選択]
      ↓
[設定ウィンドウで設定入力]
      ↓
[プラグインが以下を生成]
   - sites.geojson  （データ本体）
   - config.json         （機能設定・フィールドマッピング・圏域パラメータ）
      ↓
[プラグインが templates/base.html を読み込み]
      ↓
[出力形式を選択（タブ5で指定・デフォルトは分割ファイル方式）]
      ↓
   ┌────────────┴────────────┐
   ↓                         ↓
[分割ファイル方式（デフォルト）]   [単一HTML方式（オプション）]
geojson.js / config.js として    geojson/configを
別ファイル生成し、HTMLから        HTML内にインライン埋め込み
<script src="..."> で読み込む
   └────────────┬────────────┘
      ↓
[完成：フォルダごと配置（分割）
 または ダブルクリック（単一HTML）で開ける検索サイト]
```

**出力形式は2種類から選べるようにする**（タブ5「出力設定」で切替。**デフォルトは分割ファイル方式**、単一HTML方式はオプション設定として選択できる）。

| 方式 | 特徴 | 向いているケース |
|---|---|---|
| **分割ファイル方式（デフォルト）** | `sites.geojson.js`（またはGeoJSONをJS変数として書き出したファイル）と`config.js`を別ファイルにし、base.htmlから`<script src="...">`で読み込む | 通常の運用全般。データ件数が多い場合、データだけ後から差し替えたい場合（HTML本体はそのままにデータ更新したい等） |
| **単一HTML方式（オプション）** | GeoJSON・configをHTML内に `<script>` タグでインライン埋め込み。ファイル1つで完結。 | 「とにかく1ファイルだけ渡したい」という配布上の制約がある場合など、明示的に単一化を選んだとき |

分割ファイル方式（デフォルト）を選んだ場合、出力は「HTML1枚＋JSファイル2つ」のセットになるため、配布時はフォルダごとコピー（またはZIP化）する運用になる点を利用者向けの案内に明記する。

## 3. プラグインUI仕様（設定ウィンドウ）

既存の「施設フィルタープラグイン」（ドックパネル型、条件式ビルダー、GeoPackage出力）の**内部ロジック（属性取得・条件式ビルダー等）は流用**しつつ、UIの見せ方は変更する。

> **画面形式について**：既存の施設フィルタープラグインは「ドックパネル型」（QGISメインウィンドウの横に常駐する形）だが、本プラグインは**独立したウィンドウ（`QDialog`）として起動する形式**にする。ツールバーのアイコンやメニューから実行すると、単体のアプリケーションのようなウィンドウがポップアップし、そこで全ての設定を行う。QGIS本体のレイアウトに組み込まれないため、「ちょっとした別アプリを起動した」ような操作感になる。

> **UI実装方式について**：QGISプラグインのUIは「Qt Designerで`.ui`ファイルを見た目から作る」「PyQtコードのみで組む」のどちらでも実現できる。Qt Designerは開発時の補助ツールであり、完成後の利用者側には一切不要（インストールも操作も不要）。ドックパネル型からウィンドウ型に変わるため、既存の施設フィルタープラグインのUIコードは「ロジック部分は流用、画面の入れ物（`QDockWidget`→`QDialog`）は作り直し」という形になる。

> **多言語対応について**：公式QGIS Pluginsリポジトリへの公開を前提とするため、UI文字列は英語＋日本語の両方に対応させる（詳細は10章）。実装時は最初から `self.tr("...")` で文字列を囲んでおくこと。

### 3.1 パネル構成（タブ分け推奨）

**タブ1：データ設定（レイヤー管理）**

複数レイヤーを扱えるようにする。「検索対象レイヤー」（施設データ本体、1つ）と「参照レイヤー」（任意・複数追加可）に分けて管理する。

*検索対象レイヤー（1つ）*
- レイヤー選択コンボボックス（ロード済みレイヤーから選択。QGISのレイヤーパネル上のグループ構成をそのまま反映し、どのグループに属するレイヤーかが分かるように表示する。実装例：`QgsMapLayerComboBox`単体では階層表示できないため、`QTreeView`＋`QgsLayerTreeModel`ベースのツリー形式にする、またはコンボボックスの項目名を「グループ名 / レイヤー名」のようにインデント・階層表記して見分けられるようにする）
- 属性マッピングテーブル（QTableWidget）
  - 列：「QGIS側の属性名」「表示ラベル（日本語名）」「検索対象にするか（チェックボックス）」「一覧表示するか（チェックボックス）」
  - 既存の条件式ビルダーのフィールド一覧取得ロジックを流用可能
- 緯度経度フィールドの指定（既に緯度経度列がある場合）、またはジオメトリから自動算出するオプション

*参照レイヤー（任意・複数追加可）*

qgis2webのように、施設データ以外のレイヤー（例：行政区界、避難所、道路網等）も一緒に地図へ載せ、表示/非表示やクラスタリングを個別に設定できるようにする。

| 列 | 内容 |
|---|---|
| レイヤー名 | 地図上の凡例・レイヤーリストに表示するラベル（編集可） |
| 初期表示ON/OFF | 出力時点で最初から表示するか（チェックボックス） |
| クラスタリングON/OFF | 点データの場合のみ有効。大量の点をまとめて表示するか |
| スタイル | 4.2の仕組みでQGIS側の表示設定（色・サイズ・ラベル等）をそのまま継承（個別設定は不要） |

「レイヤーを追加」ボタンでロード済みレイヤーの一覧（グループ構成込み）から選び、テーブルに行として追加していく。検索対象レイヤーとは異なり、参照レイヤーは検索・フィルタ機能の対象にはならず、あくまで地図上の表示/非表示切り替えの対象。

**タブ2：表示設定**

qgis2webで設定できる基本的な地図表示項目を標準搭載する。

*画面サイズ・レスポンシブ*
- 表示範囲：「画面いっぱいに表示（フルスクリーン相当）」／「固定サイズで表示（幅×高さをピクセルで指定）」のいずれかを選択
- スマートフォン対応（レスポンシブ）：ON/OFFのチェックボックス（**デフォルトON**）。ONの場合、画面幅に応じてレイアウトが自動調整される（viewportメタタグ・CSSメディアクエリで対応）

*初期表示位置・ズーム*
- 初期表示：「データの範囲に自動フィット」／「中心座標・ズームレベルを手動指定」のいずれかを選択
- 手動指定の場合、緯度・経度・初期ズームレベルの入力欄を表示
- 最小ズームレベル・最大ズームレベル（スピンボックス）

*表示ウィジェット（個別にON/OFF）*

| ウィジェット | 説明 |
|---|---|
| スケールバー | 地図上に縮尺を表示 |
| 現在地取得（Geolocate） | ブラウザの位置情報を使い、現在地を地図上に表示 |
| フルスクリーン切替ボタン | ブラウザのフルスクリーン表示に切り替えるボタン |
| 計測ツール | 距離・面積を地図上で計測する機能 |
| 帰属表示（Attribution） | 出典・著作権表記のテキスト（自由入力） |
| レイヤーリスト（表示/非表示切替パネル） | 地図上に検索対象レイヤー・参照レイヤーの一覧を表示し、チェックボックスで表示/非表示を切り替えられるようにする（qgis2webの「Layers list」相当。**デフォルトON**） |

**タブ3：機能設定**
機能ごとにON/OFFのチェックボックスを並べる。

| 機能 | 説明 | 追加設定 |
|---|---|---|
| 検索ボックス | キーワード・部分一致検索 | 検索対象フィールド（タブ1で指定） |
| 条件フィルタ | 施設種別等での絞り込み | フィルタ対象フィールド選択 |
| マーカークラスタリング | 大量データの表示最適化 | クラスタ半径 |
| ヒートマップ | 密度可視化 | 半径・強度 |
| 圏内施設抽出（半径検索） | 地図上の任意地点、または既存施設を中心に、指定した半径以内の施設を一覧抽出する汎用機能 | 中心の指定方法（クリック地点／施設選択）、デフォルト半径、種別での絞り込みの有無 |
| 圏域カバー分析 | 施設種別ごとの圏域を描画・カバー率表示 | 種別ごとの圏域距離（タブ4で自由追加） |
| 表示中施設一覧表示 | 検索・フィルタ後に地図上へ表示されている施設をサイドパネルに一覧表示。項目クリックで地図がその施設にパン/ズーム | 一覧に表示する項目（タブ1のlistFields）、並び替え順（名前順／距離順等） |
| リスト書き出し/読み込み | 検索結果のCSV等出力 | - |

### 3.1.1 UIレイアウト配置設定

検索ボックス・フィルタ・施設一覧・圏域設定パネルなど、画面上のUI要素をどこに配置するかを**要素ごとに個別に選べる**ようにする。

**配置の選択肢**

| 配置先 | 説明 |
|---|---|
| 上部（top） | 地図の上に横並びで配置 |
| 下部（bottom） | 地図の下に横並びで配置 |
| 左側（left） | 地図の左にサイドパネルとして配置 |
| 右側（right） | 地図の右にサイドパネルとして配置 |
| 非表示（hidden） | その要素自体を出力に含めない |

各UI要素（検索ボックス／条件フィルタ／施設一覧／圏域設定パネル等）は、それぞれ独立して配置先を選べる。例えば「検索ボックスは上部、施設一覧は右側、圏域設定パネルは左側」のように**同時に複数箇所へ分割配置**できる。同じ配置先を選んだ要素同士は、その領域内で縦（または横）に積み重ねて表示する。

**プレビュー表示**

設定ウィンドウ内に、選択中のレイアウトを簡易的に確認できるプレビュー領域を設ける。

- 地図領域を表す1つの大きな灰色の四角と、その周囲（上下左右）に配置候補の帯（バー）を描いた簡易図とする
- 各UI要素の設定を変更すると、対応する帯の中にその要素名（例：「検索」「一覧」等）のラベルが表示され、リアルタイムに反映される
- 実際の見た目そのものを精密に再現する必要はなく、あくまで「どこに何を置いたか」を一目で把握できれば十分（実装は`QPainter`でシンプルな矩形とテキストラベルを描画する程度でよい。凝ったレンダリングは不要）

**config.jsonへの反映**

```json
"layout": {
  "search": "top",
  "filter": "top",
  "pointList": "right",
  "coverageControls": "left"
}
```

- キーはUI要素の識別子、値は配置先（`top` / `bottom` / `left` / `right` / `hidden`）
- テンプレートHTML側は、この値に応じてCSSクラス（例：`.panel-top` `.panel-right`）を要素に付与し、CSS側のレイアウト（flexboxやgrid）で実際の配置を切り替える

**タブ4：圏域パラメータ設定**

施設種別ごとの圏域距離を編集できるテーブル。公園・消防署等の種別ごとに固定の初期値は持たせず、**自由追加のみ**とする。

| 施設種別 | 圏域距離 | 備考 |
|---|---|---|
| （自由追加） | ユーザー入力 | 種別フィールドの値と紐付け |

種別フィールドの値（タブ1の属性マッピングで指定した「種別」列の実際の値）を読み取り、行の追加候補として選択肢に出す（QGISレイヤーの属性値をユニーク抽出）。ユーザーは対象の種別を選び、圏域距離を自分で入力して行を追加する。距離が複数段階（例：250m/500m/1km）必要な場合は、カンマ区切り等で複数値を1行に入力できるようにする。

**タブ5：出力設定**
- タイトル・見出しテキスト入力（生成HTMLの `<title>` やヘッダーに反映）
- **出力形式の選択（ラジオボタン、デフォルト：分割ファイル）**：「分割ファイル（HTML＋GeoJSON/config別ファイル）」／「単一HTMLファイル（オプション）」
- 出力先パス選択（分割ファイル方式：出力先フォルダ／単一HTML方式：ファイルパス）
- 「HTML生成」実行ボタン
- プログレスバー（大量データの場合の書き出し待ち表示用）

### 3.2 設定の保存・読み込み

- 設定内容（タブ1〜5すべて）をプラグイン内で `.json` として保存・読み込みできるようにする（毎回設定し直さなくて済むように）。
- 「設定を保存」「設定を読み込む」ボタンをパネル下部に配置。

## 4. データ設計（config.json スキーマ）

プラグインが生成し、HTMLテンプレート側のJSが読み込む設定ファイルの構造案。

```json
{
  "meta": {
    "title": "○○市 公共施設検索システム",
    "generatedAt": "2026-07-18T00:00:00+09:00"
  },
  "uiLanguage": {
    "default": "ja",
    "available": ["ja", "en"],
    "titleEn": "○○ City Public Facility Search"
  },
  "display": {
    "sizeMode": "fullscreen",
    "fixedSize": { "width": 960, "height": 640 },
    "responsive": true,
    "initialView": { "mode": "autoFit" },
    "minZoom": 5,
    "maxZoom": 19,
    "widgets": {
      "scaleBar": true,
      "geolocate": false,
      "fullscreenButton": true,
      "measureTool": false,
      "attribution": "○○市 提供データ"
    },
    "layerControl": {
      "enabled": true
    }
  },
  "referenceLayers": [
    {
      "id": "ref_wards",
      "label": "行政区界",
      "defaultVisible": true,
      "clustering": false,
      "geojsonKey": "ref_wards"
    }
  ],
  "fields": {
    "idField": "site_id",
    "nameField": "site_name",
    "typeField": "site_type",
    "latField": "lat",
    "lngField": "lng",
    "searchFields": ["site_name", "address"],
    "listFields": [
      { "key": "site_type", "label": "種別" },
      { "key": "address", "label": "住所" },
      { "key": "capacity", "label": "定員" }
    ]
  },
  "features": {
    "search": true,
    "filter": true,
    "clustering": true,
    "heatmap": false,
    "radiusSearch": {
      "enabled": true,
      "defaultRadius": 500,
      "centerMode": "clickOrSite",
      "typeFilterEnabled": true
    },
    "coverageAnalysis": {
      "enabled": true,
      "radiusByType": {
        "（種別名の例1）": [250, 500, 1000],
        "（種別名の例2）": [2000]
      }
    },
    "pointList": true,
    "exportList": true
  },
  "layout": {
    "search": "top",
    "filter": "top",
    "pointList": "right",
    "coverageControls": "left"
  }
}
```

- `features` 配下は、タブ3のチェックボックスと1:1対応させる。
- `uiLanguage` は出力サイトのUIラベル日英切替設定（10.3参照）。`default`は初回表示時の言語、`titleEn`はタイトルの英語表記。UI文字列本体は`js/ui-i18n.js`内の辞書（`{ ja: {...}, en: {...} }`）で管理し、config.json自体には辞書は持たせない（タイトル等、config側で個別指定が必要なものだけここに置く）。
- `display` はタブ2（表示設定）の内容がそのまま反映される。`sizeMode`が`fixed`の場合のみ`fixedSize`を参照する。`initialView.mode`が`manual`の場合は`center`（緯度経度）と`zoom`を追加で持つ。
- `referenceLayers` はタブ1で追加した参照レイヤーの配列。`geojsonKey`は、そのレイヤーのGeoJSONデータを分割ファイル方式で読み込む際のファイル識別に使う（4.3参照）。検索対象レイヤー（`sitesData`）とは別に扱う。
- `layout` は3.1.1で説明したUI要素の配置設定（`top`/`bottom`/`left`/`right`/`hidden`）が、タブ3の配置設定パネルで選んだ内容そのまま反映される。
- `radiusByType` は種別名をキーにした辞書。タブ4で**ユーザーが自由に追加した種別・距離のペアがそのまま反映される**（ソフト側が公園・消防署等の固定の初期値を持つわけではない。上記スキーマ例の「（種別名の例1）」等は、タブ4で何かを追加した場合のイメージを示しているだけで、実際のキー名・値はユーザーの入力次第）。テンプレート側は、ここに登録のない種別の施設には圏域表示を行わない（またはUI上で圏域なしとして扱う）。
- 人口メッシュ重ね合わせ機能は**実装見送り**（将来検討事項。8章参照）のため、`features`には含めていない。

## 4.1 GeoJSON側の要件

- 属性名はconfig.jsonの `fields` で指定したキー名と一致させる（プラグイン側でリネームして出力するか、config側でマッピングするかは実装時に決定。**推奨：出力時に属性名を統一済みGeoJSONにしてしまう方がテンプレート側のコードがシンプルになる**）。
- 座標系はWGS84 (EPSG:4326) に変換して出力（Leafletの前提に合わせる）。
- 検索対象レイヤー（施設データ本体）は`sites.geojson`として出力し、これまで通り検索・フィルタ等の対象になる。
- 参照レイヤー（タブ1で追加したもの）は、レイヤーごとに個別のGeoJSONとして出力する（例：`ref_wards.geojson`）。ファイル名・識別子はconfig.jsonの`referenceLayers[].geojsonKey`と対応させる。
- ラベル表示用に、4.2.1.1で説明する**計算済みラベル文字列を`label_text`属性として全フィーチャに付与**する（単一フィールド表示・式のどちらの場合も統一してこの属性を持たせる）。

## 4.2 スタイル情報の書き出し（QGISレイヤー表示設定の継承）

**方針**：出力される検索サイトの見た目（マーカー・ラベル）は、QGIS上でそのレイヤーに設定されている表示設定（シンボロジー）を**そのまま継承する**。qgis2webのような「レイヤーの見た目をそのまま持っていく」という利点は引き継ぎつつ、検索・フィルタ等のアプリ機能を組み合わせる。

### 4.2.1 抽出する表示設定の項目

| 項目 | QGIS側の設定元 | 抽出するプロパティ |
|---|---|---|
| マーカー（点シンボル） | `QgsMarkerSymbol` | 色、サイズ、形状（円・四角・アイコン等）、不透明度 |
| ライン | `QgsLineSymbol` | 線の太さ、色、線種（実線・破線等） |
| ポリゴン | `QgsFillSymbol` | 塗りつぶし色、輪郭線の色・太さ |
| ラベル（注記） | `QgsPalLayerSettings` / `QgsTextFormat` | フォント名、サイズ、色、太字・斜体、バッファ（縁取り）の色・太さ、配置位置、**ラベルの文字内容（単一フィールド、または式）** |

> **ラベルの文字内容について**：QGISのラベル設定は「単一フィールドを表示」または「式を使用（`isExpression = True`）」のどちらかで構成されている。qgis2webは単一フィールドの表記にしか対応していなかったが、本プラグインは**式（例：`"施設名称" || ',' || "築年" || '年' || ',' || "延床面積" || '㎡'`）にも対応する**（詳細は4.2.1.1）。

#### 4.2.1.1 ラベル式（Expression）への対応方法

**方針**：JavaScript側で式を再評価するのではなく、**QGIS側（Python）で式をあらかじめ評価し、結果の文字列をGeoJSONの属性として書き出す**。QGISの`QgsExpression`エンジンをそのまま使えるため、対応できる式の範囲がqgis2webより大幅に広がり、JS側の実装もシンプルになる。

**処理の流れ**
1. レイヤーの`labeling()`（`QgsVectorLayerSimpleLabeling`）から`QgsPalLayerSettings`を取得
2. `settings.isExpression`が`True`の場合、`settings.fieldName`に式の文字列が入っているので、これを`QgsExpression`として読み込む
3. 各フィーチャごとに`QgsExpressionContext`（レイヤー・フィーチャのスコープを設定したもの）を使って式を評価し、結果の文字列を取得（`expression.evaluate(context)`）
4. 評価結果を新しい属性（例：`label_text`）としてフィーチャに追加し、GeoJSON出力に含める
5. `isExpression`が`False`（単一フィールド表示）の場合は、そのフィールドの値をそのまま`label_text`として複製する（式の有無にかかわらず、テンプレート側は常に`label_text`という統一された属性だけを見ればよいようにする）

**config.jsonへの反映**

```json
"fields": {
  "labelField": "label_text"
}
```

- `labelField`はGeoJSON側で常に用意される「表示用に計算済みのラベル文字列」を指す。テンプレート側（Leaflet）は式の中身を意識せず、この属性をそのまま表示するだけでよい。
- 改行を含めたい場合、QGIS側の式で`\n`や`wordwrap()`関数を使えば、その通りの改行がそのまま`label_text`に反映される（CSS側は`white-space: pre-line`等で改行を保持する）。

### 4.2.2 レンダラーの種類への対応

QGISレイヤーの描画方式（レンダラー）によって抽出方法が変わるため、対応範囲を明確にする。

- **単一シンボル（Single Symbol Renderer）**：レイヤー全体が1つのスタイル → そのまま1つのスタイル定義として抽出（対応必須）
- **分類シンボル（Categorized Renderer）**：施設種別ごとに色や形状が分かれているケースに対応（例：公園は緑の丸、消防署は赤い四角）→ 種別の値ごとにスタイルを抽出し、種別フィールドの値をキーにしたスタイル辞書として書き出す（**施設データでは種別ごとに色分けしているケースが多いと想定されるため、優先的に対応**）
- **段階分け・ルールベース等の高度なレンダラー**：フェーズ1〜3では対応を見送り、将来検討とする（8章参照）

### 4.2.3 style.json のスキーマ例

```json
{
  "defaultStyle": {
    "marker": { "color": "#4a4a4a", "size": 8, "shape": "circle", "opacity": 0.9 },
    "label": {
      "fontFamily": "Noto Sans JP",
      "fontSize": 12,
      "color": "#333333",
      "bold": false,
      "buffer": { "color": "#ffffff", "width": 2 }
    }
  },
  "byCategory": {
    "site_type": {
      "公園": {
        "marker": { "color": "#4caf50", "size": 8, "shape": "circle" },
        "label": { "fontFamily": "Noto Sans JP", "fontSize": 12, "color": "#2e7d32", "buffer": { "color": "#ffffff", "width": 2 } }
      },
      "消防署": {
        "marker": { "color": "#e53935", "size": 10, "shape": "square" },
        "label": { "fontFamily": "Noto Sans JP", "fontSize": 12, "color": "#b71c1c", "buffer": { "color": "#ffffff", "width": 2 } }
      }
    }
  }
}
```

- `defaultStyle`：単一シンボルの場合、またはカテゴリに一致しない場合のフォールバック
- `byCategory`：分類シンボルの場合、フィールド名（例：`site_type`）をキーにして、値ごとのスタイルを保持

### 4.2.4 実装方法（PyQGIS側）

1. 対象レイヤーの`renderer()`を取得し、`QgsSingleSymbolRenderer`か`QgsCategorizedSymbolRenderer`かを判定
2. シンボルから`symbolLayer(0)`等を辿り、色（`.color()`）・サイズ（`.size()`）・形状（`.shape()` または `QgsSimpleMarkerSymbolLayerBase.shapeToName()`）を取得しHEXカラー等に変換
3. ラベル設定は`layer.labeling()`（`QgsVectorLayerSimpleLabeling`）から`QgsPalLayerSettings`→`.format()`（`QgsTextFormat`）を取得し、フォント名・サイズ・色・バッファ設定を読み出す
4. ラベルの文字内容は4.2.1.1の手順で`isExpression`を判定し、式の場合は`QgsExpression`で各フィーチャごとに評価、単一フィールドの場合はその値をそのまま使い、いずれも`label_text`属性としてGeoJSON出力用のフィーチャに追加する
5. 取得した値をPythonの辞書に詰めてJSONとして書き出す

### 4.2.5 テンプレート側（Leaflet）での適用方法

- マーカー：Leafletの`L.circleMarker`または`L.divIcon`を使い、style.jsonの色・サイズをインラインスタイルとして適用
- ラベル：各フィーチャの`label_text`属性（4.2.1.1でQGIS側にて評価済みの文字列）を、`L.marker`のツールチップ/permanent labelにそのまま表示する。JS側で式を評価・組み立てる処理は不要。CSSで`font-family` `font-size` `color` `text-shadow`（バッファ効果の疑似再現）を適用し、複数行の場合は`white-space: pre-line`で改行を保持する
- カテゴリ分けがある場合は、各施設の`site_type`の値を見て`byCategory`から該当スタイルを引き、なければ`defaultStyle`を使う

## 4.3 参照レイヤーの扱い

- 参照レイヤーは検索・フィルタ・一覧表示の対象にはならない。地図上への表示/非表示切り替えとクラスタリング（点データの場合）のみを提供する。
- スタイルは4.2の仕組みで検索対象レイヤーと同様に抽出し、`style.json`内に参照レイヤーごとのエントリを持たせる（例：`style.json`のトップレベルに`bySite`とは別に`byReferenceLayer`のような区分を設ける）。
- ポリゴン・ラインの参照レイヤー（行政区界等）はクラスタリング設定自体が無効（クラスタリングは点データのみに適用）。

## 5. HTMLテンプレート構成

`osaka-facilities` の実装を機能単位でモジュール化し、config.jsonの値でON/OFFする形に整理する。

### 5.0 デザイン方針

配布用テンプレートのため、見た目は**シンプル・最低限**にする。

- ダークモードは実装しない（機能としても、切替UIとしても不要）
- 配色は**薄いグレーを基調**とする（背景・パネル・境界線などに薄いグレー系の色を使い、主張の強い配色は避ける）
- 装飾よりも「情報の見やすさ・操作のわかりやすさ」を優先する
- **色やサイズはCSS変数（`:root { --bg-color: ...; --border-color: ...; }` 等）にまとめ、`style.css`の先頭にひとまとめに記述する。** これにより、作成した本人が後からコードを直接編集して配色・余白等を調整したい場合に、CSS変数の値を書き換えるだけで済むようにする（HTML/JSの各所に色コードが散らばらないようにする）
- 主要なCSSブロックには簡単な**英語コメント**を添え、どこを変えれば何が変わるかが後から見て分かるようにする（QGIS公式リポジトリの審査は英語で行われるため、コード内コメントも英語で統一する）

### 5.1 ファイル構成（開発時。最終的には1ファイルに結合）

```
template/
 ├─ base.html          … 骨格（head, body, 埋め込み箇所のプレースホルダ）
 ├─ css/style.css      … スタイル（シンプル・最低限。薄いグレー基調の配色）
 ├─ js/map-core.js     … 地図初期化・マーカー描画（必須コア）
 ├─ js/style-renderer.js … style.jsonに基づくマーカー・ラベルのスタイル適用
 ├─ js/display-settings.js … 画面サイズ・レスポンシブ・初期表示位置・ズーム制限の適用
 ├─ js/map-widgets.js  … スケールバー・現在地取得・フルスクリーン・計測ツール・帰属表示
 ├─ js/layer-control.js … レイヤーリスト（表示/非表示切替パネル）、参照レイヤーの読み込み・クラスタリング
 ├─ js/search.js       … 検索・フィルタ機能
 ├─ js/clustering.js   … マーカークラスタリング
 ├─ js/heatmap.js      … ヒートマップ
 ├─ js/radius-search.js … 圏内施設抽出（半径検索、ハバーサイン距離計算）
 ├─ js/coverage.js     … 圏域カバー分析（バッファ円描画）
 ├─ js/point-list.js   … 表示中施設一覧表示
 ├─ js/layout.js        … config.layoutに応じたUI配置の適用
 ├─ js/ui-i18n.js       … 出力HTML側のUIラベル日英切替（プラグイン自体の多言語対応とは別）
 └─ js/export.js       … リスト書き出し/読み込み
```

### 5.2 各JSモジュールの起動条件

各モジュールの初期化関数は、`config.features.xxx === true` の場合のみ呼び出す。例：

```javascript
// main.js（結合後の起点）
applyDisplaySettings(map, config.display); // 画面サイズ・レスポンシブ・初期表示位置・ズーム制限（常時実行）
initMapWidgets(map, config.display.widgets); // スケールバー・現在地取得・フルスクリーン・計測ツール・帰属表示
initLayerControl(map, config.referenceLayers, config.display.layerControl); // レイヤーリスト・参照レイヤー読み込み
renderSitesWithStyle(map, sitesData, styleData); // style.jsonに基づきマーカー・ラベルを描画（常時実行、機能ON/OFFの対象外）
if (config.features.search) initSearch(config, sitesData);
if (config.features.clustering) initClustering(map, sitesData);
if (config.features.heatmap) initHeatmap(map, sitesData);
if (config.features.radiusSearch.enabled) initRadiusSearch(config, sitesData);
if (config.features.coverageAnalysis.enabled) initCoverage(config, sitesData);
if (config.features.pointList) initPointList(config, sitesData);
if (config.features.exportList) initExport(sitesData);
applyLayout(config.layout); // UI要素の配置をconfigに従って適用（常時実行）
```

### 5.3 結合（バンドル）方法

- プラグイン側でPythonの単純な文字列結合（またはテンプレートエンジン、例：`string.Template` や `Jinja2` があれば利用）で処理する。出力形式（デフォルト：分割ファイル／オプション：単一HTML）によって最後の埋め込み・書き出し方だけを切り替える。

**共通処理**
  1. `base.html` を読み込み
  2. `<!-- INJECT_SCRIPTS -->` の位置に、有効な機能のJSファイル内容をすべて `<script>` タグ内にまとめて挿入

**分割ファイル方式の場合（デフォルト）**
  3. `config.js`（`const config = {...};` を含むファイル）を別途生成
  4. `sites.js`（`const sitesData = {...};` を含むファイル）を別途生成
  5. `style.js`（`const styleData = {...};` を含むファイル、4.2のstyle.json相当）を別途生成
  6. `base.html` の該当箇所には `<script src="config.js"></script>` `<script src="sites.js"></script>` `<script src="style.js"></script>` を挿入
  7. HTML・config.js・sites.js・style.jsの4ファイルを同一フォルダに書き出し

**単一HTML方式の場合（オプション選択時）**
  3. `<!-- INJECT_CONFIG -->` の位置に `const config = {...};` を直接埋め込み
  4. `<!-- INJECT_DATA -->` の位置に `const sitesData = {...};` を直接埋め込み
  5. `<!-- INJECT_STYLE -->` の位置に `const styleData = {...};` を直接埋め込み
  6. 1ファイルとして出力先パスに書き出し

- Jinja2が庁内環境で使えない場合を考慮し、**まずは標準ライブラリのみで完結する文字列置換方式**を第一候補とする（インストール制限のある庁内環境向け）。

## 6. 開発ステップ（フェーズ分け）

### フェーズ1：最小構成（MVP）
- タブ1（データ設定：検索対象レイヤーのみ、参照レイヤーは対象外）＋タブ2（表示設定のうち、表示範囲・スマホ対応・初期表示位置のみ）＋タブ5（出力設定）を実装
- 機能は「検索」「一覧表示」のみ固定でON
- **4.2のスタイル抽出（まずは単一シンボルのみ対応）を実装し、GeoJSON・config.jsonに加えてstyle.jsonも生成する。ラベルの文字内容抽出（4.2.1.1、単一フィールド／式の両方）もこの時点で実装する**（`QgsExpression`を使えば式評価は`.evaluate()`一発で済み、単一フィールド対応と実装コストがほぼ変わらないため、最初から両対応にする）
- GeoJSON生成＋config.json・style.json生成＋base.html結合＋出力、の一連の流れを通す

### フェーズ2：機能選択式への拡張
- タブ3（機能設定）を実装し、既存のosaka-facilities機能（クラスタリング・ヒートマップ・圏域分析）をモジュール化してテンプレートに組み込む。あわせて、新規機能として「圏内施設抽出（半径検索）」「表示中施設一覧表示」を汎用モジュールとして追加する
- スタイル抽出を**分類シンボル（Categorized Renderer）対応**に拡張し、施設種別ごとに色分けされたレイヤーを正しく継承できるようにする
- タブ1に**参照レイヤー管理**（複数追加・表示/非表示・クラスタリング）を実装
- タブ2の**表示ウィジェット**（スケールバー・現在地取得・フルスクリーン・計測ツール・帰属表示）と**レイヤーリストパネル**を実装

### フェーズ3：圏域パラメータのGUI化
- タブ4を実装し、種別ごとの圏域デフォルト値を編集可能に

### フェーズ3.5：UIレイアウト配置設定・プレビュー
- 3.1.1のUIレイアウト配置設定（要素ごとの上下左右配置・分割配置）を実装
- 設定ウィンドウ内の簡易プレビュー描画（`QPainter`による矩形＋ラベル表示）を実装
- テンプレート側は、configの`layout`値に応じてCSSクラスを切り替える仕組み（`js/layout.js`）を実装

### フェーズ4：設定の保存・読み込み、UI仕上げ
- 設定JSONの保存/読み込み
- プログレスバー、エラーハンドリング（属性未設定時の警告等）
- 出力HTML側のUIラベル日英切替（`js/ui-i18n.js`、言語切替ボタン）を実装（10.3、必須要件）

## 10. 多言語対応・公開ドキュメントの言語方針

公式QGIS Pluginsリポジトリへの公開を前提とするため、以下の言語方針を設定する。

### 10.1 ドキュメント（説明文）の言語

QGIS Pluginsリポジトリのページ・README・ヘルプ文書などの「説明書き」は、**基本は英語**で作成する。日本語版も用意できると望ましい（併記、または別ファイルとして用意）。

| ドキュメント | 言語方針 |
|---|---|
| `metadata.txt` の `description` / `about` | **英語（必須）**。QGIS公式リポジトリの審査・検索対象になるため英語が基本 |
| README（GitHub等） | 英語をメインに、日本語セクションを併記（例：`## English` / `## 日本語` で分ける） |
| プラグインのヘルプページ・使い方ガイド | 英語版を主とし、日本語版を別ファイル（`help_en.md` / `help_ja.md`）として用意 |
| この設計書自体 | 現状どおり日本語のまま（内部の設計・指示用途のため） |

### 10.1.1 コード内コメントの言語

プラグイン本体・テンプレートHTML/JS/CSS内の**コードコメントは英語で統一**する（QGIS公式リポジトリの審査担当者が英語で読むことを前提とするため）。日本語話者向けの補足が必要な場合は、この設計書（日本語）側に書き、コード中には残さない。

### 10.2 プラグイン本体（UI・メッセージ）の言語

プラグインのUI（ボタン・ラベル・メッセージ等）は**英語＋日本語の両方に対応**させる。QGISプラグインの標準的な多言語対応の仕組み（Qt Linguist方式）に沿って進める。

**実装方法**
1. UI文字列はコード中で直接書かず、必ず `self.tr("...")` で囲む（Qtの翻訳対象として認識させるため）。例：`self.tr("Select Layer")`
2. 開発時のベース言語は英語にする（`self.tr("Layer selection")` のように英語で記述）
3. `pylupdate5`（またはQGIS Plugin Builderに付属のツール）でコードから翻訳対象文字列を抽出し、`i18n/facility_app_generator_ja.ts` を生成
4. Qt Linguist（またはテキストエディタ）で `.ts` ファイルに日本語訳を追記
5. `lrelease` で `.ts` → `.qm`（コンパイル済み翻訳ファイル）に変換
6. プラグイン起動時にQGISの表示言語設定を見て、日本語環境なら日本語の `.qm` を自動読み込みする処理を入れる（QGIS本体の言語切り替えの仕組みに準拠）

**ファイル構成への追加**

```
plugin/
 ├─ i18n/
 │   ├─ facility_app_generator_ja.ts   … 翻訳ソース（日本語）
 │   └─ facility_app_generator_ja.qm   … コンパイル済み翻訳（配布物に含める）
 ├─ metadata.txt   … description/about は英語で記述
 └─ （既存のプラグイン本体ファイル）
```

### 10.3 生成される検索サイト（出力HTML）側の言語

出力される施設検索サイト自体（`base.html`）は、庁内向け・住民向けの実務ツールであるため、施設名・住所などのデータ自体は**日本語のまま**でよい。一方、**UIラベル部分（検索ボックスのプレースホルダ、ボタン名、見出し等）は日本語／英語の切替を必須要件とする**。実装方法は10.2と同様、UI文字列を`data-i18n`属性等で辞書参照する簡易な仕組みにし、ページ上部に言語切替ボタン（例：「JP / EN」）を常設する。データ本体（施設名・住所等）の翻訳までは対象外とし、あくまで画面のラベル・案内文の切替にとどめる。

### 10.4 進め方の目安

- フェーズ1（MVP）の実装時点から `self.tr()` を使う書き方を徹底しておくと、後から一括で翻訳対応する際の手戻りが少ない
- 翻訳ファイル（`.ts`/`.qm`）の作成は、機能が固まってきたフェーズ2〜3あたりでまとめて行う方が効率的（文字列が頻繁に変わる間は翻訳のやり直しが発生するため）
